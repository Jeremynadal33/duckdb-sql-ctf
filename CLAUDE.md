# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

A CTF (Capture The Flag) project introducing DuckDB by learning to query multiple database engines. The infrastructure provisions a PostgreSQL 16 RDS instance on AWS that serves as one of the query targets.

## Terraform Commands

All Terraform commands run from the `terraform/` directory. The root `.env` file provides AWS credentials only.

```bash
# Load AWS credentials before any terraform command
source .env

# Initialize (required after backend/provider changes)
terraform init

# Format, validate, plan, apply
terraform fmt
terraform validate
terraform plan
terraform apply

# Connect to the database locally (master password is generated by Terraform and stored in SSM)
psql "host=$(terraform output -raw db_endpoint | cut -d: -f1) port=5432 dbname=$(terraform output -raw db_name) user=$(terraform output -raw pg_user) password=$(aws ssm get-parameter --name "$(terraform output -raw ssm_pg_master_password)" --with-decryption --query Parameter.Value --output text)"
```

## Architecture

- **Infrastructure**: Terraform on AWS (eu-west-1), state stored in S3 (`s3://ippon-data-dev-tfstates-bucket/duckdb-sql-ctf/`)
- **Database**: PostgreSQL 16 on RDS (`db.t4g.micro`), publicly accessible on port 5432, inside a custom VPC with two public subnets
- **Terraform layout**: `provider.tf` (backend + provider + default tags), `network.tf` (VPC/subnets/IGW/routes), `security.tf` (SG), `database.tf` (RDS), `variables.tf`, `outputs.tf`
- **Credentials**: `pg_user`/`pg_ro_user` are Terraform variables (defaults `ctfadmin`/`ctfplayer`); the master and read-only passwords are generated via `random_password` (`secrets.tf`) and stored as SecureString SSM parameters under `/${bucket_name}/pg/`. Only AWS credentials come from `.env`.

## Flag Logic (dual maintenance required!)

Flag construction logic lives in **two places that must stay in sync**:

1. **Python generators** (`data_generator/src/data_generator/generators/`) — build flags and embed them in generated data (JSON logs, Parquet metadata, Postgres rows)
2. **Terraform answer files** (`terraform/locals.tf` flags + `terraform/storage.tf` S3 objects) — upload expected flags to `s3://bucket/leaderboard/answers/scenario_{N}.txt` for the Lambda answer checker

When modifying flag format or content for any scenario, **update both places** or the answer checker will reject correct submissions.

### Scenario 0 — Tutorial (no points)

Scenario 0 ("00 — Tutoriel") is a non-scored warm-up. Its flag is a fixed string kept in sync in
**two** places: `SCENARIO0_FLAG` in `data_generator/src/data_generator/constants.py` (embedded in
`data/buildings.csv` via `generators/scenario0_csv.py`) and `flag_scenario0` in `terraform/locals.tf`
(uploaded as `leaderboard/answers/scenario_0.txt` by `terraform/storage.tf`).

It introduces a **third synchronization point**: `docs/data-agent.js` excludes scenario 0 from scoring
(the `ev` CTE in `_queryAll` filters `scenario <> 0`). The scenario-0 submission stays visible in the
activity feed but earns no points and adds no leaderboard column. If you change the scenario numbering,
keep that filter and the `allSolved` threshold (`docs/scenarios/scenarios.js`, computed from the max real
scenario number) consistent.
