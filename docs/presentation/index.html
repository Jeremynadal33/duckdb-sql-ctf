Pourquoi DuckDB ?

  • Performance exceptionnelle sur les requetes analytiques
  • Simplicite : un seul fichier, zero config (avant DuckLake au début en tout cas)
  • Multi-format reading : Parquet, CSV, JSON, Postgres, Delta, Iceberg, GSheets, ...
  • Fonctionne partout : Python, R, Node, CLI, WASM
  • Ideal pour l'exploration et le prototypage, mais pas que !

Comment on vous note

Deux modes disponibles, un malus commun

Mode « Rang »

Ordre d'arrivée par scénario

  • 1er → 100 pts
  • 2e → 90 pts · 3e → 80 pts · ...
  • 10e et au-delà → 50 pts (plancher)

Mode « Vitesse »

Temps passé sur le scénario
Δ = solved_at(N) − solved_at(N−1)

  • Plus rapide du scénario → 100 pts
  • Décroissance linéaire sur 10 min
  • Au-delà de +10 min → 50 pts

Scénario 1 : ancre = ton inscription.

Malus « Hints »

Selon Δhint = hint_ouvert − début_du_solve, appliqué sur les deux modes.

  • Δ < 3 min → −15 pts
  • Δ < 6 min → −10 pts
  • Δ < 15 min → −5 pts
  • Δ ≥ 15 min → 0 (gratuit)

Hints ouverts après solve : ignorés. Plancher final : 0 pts.

Exemple de soumission d'un flag

COPY (
    SELECT '<pseudo>' AS pseudo, <N> AS scenario, 'FLAG{...}' AS flag
) TO 's3://<bucket>/user-inputs/<pseudo>_scenario_<N>.parquet' (FORMAT PARQUET);
